# VALIDATION POLICY

教材の正確性を担保するための必須ルール。すべての教材（research/knowledge/curriculum）はこのポリシーに従う。

## 必須ルール

```text
AIの一般知識だけで教材を作ってはいけない
公式資料・README・実機確認の根拠を持つ
未確認UI操作は教材に入れない
存在しないボタン名を使わない
Windows前提と矛盾させない
```

### AIの一般知識だけで教材を作ってはいけない

「一般的にGitHubはこう使われている」という記憶ベースの知識だけでは教材を作らない。必ず `research/` に一次資料（公式README・公式Docs・実機確認メモ）を残し、そこから教材を組み立てる。

### 公式資料・README・実機確認の根拠を持つ

すべての事実（`knowledge/*/facts.md`）は、`research/`内の具体的な資料に出典を遡れる状態にする。出典のない事実は書かない。

### 未確認UI操作は教材に入れない

「おそらくこのボタンを押せば良い」という推測でのUI操作手順は禁止する。UI操作を教材（特にworkshop）に含める場合は、実機で確認した記録（`research/.../verification.md`）が存在することを前提とする。実機確認ができていない段階では、教材は概念説明（`guided_observation`など）に留める。

### 存在しないボタン名を使わない

UIは頻繁に変わる。確認していないボタン名・メニュー名・画面名を教材に書いてはいけない。過去のバージョンの記憶や他サービスとの混同にも注意する。

### Windows前提と矛盾させない

このプロジェクトは当面Windows専用である。Macのショートカット・Mac特有の操作・「Macの場合は」という分岐を教材に含めない。

## 必須Veto条件（再掲）

以下のいずれかに該当する教材は不合格とする。評価スコアがどれだけ高くても合格にしない。

```text
公式資料・実機根拠がない
実際のUI名称と違う
存在しないボタンを押させている
Windows前提と矛盾している
クイズの正解が一意でない
Workshopの成功判定が曖昧
初心者が次に何をすればいいか分からない
```

## 検証プロセスとの関係

- 機械的な整合性チェック（必須ファイルの有無、JSON構文、必須項目の有無）は `scripts/validate-curriculum.ps1` が行う。
- 内容の正確性・初心者への分かりやすさ・Veto条件への抵触は、`eval/` のcriteria・tasksを使ったeval-loopで検証する（詳細は [EVAL_LOOP_GUIDE.md](EVAL_LOOP_GUIDE.md)）。

両方を通過するまで、教材の`status`を`published`にしない。
