# LESSON AUTHORING GUIDE

`lesson.md` を書くときの執筆ガイドライン。

## 前提

- 読者はPC初心者・IT初心者・GitHub初心者。専門用語は初出時に必ず説明する。
- 対象OSはWindowsのみ。Windows以外の操作・画面には触れない。
- lesson.mdの内容は `knowledge/<theme>/<ID>/facts.md` の範囲を超えない。factsにない話を「たぶんこうだろう」で補わない。

## 構成の型

1. **導入（なぜこれを学ぶか）**: 1〜2文で、この教材が何の役に立つかを示す。
2. **本文（事実の説明）**: factsに書かれた事実を、初心者にも分かる順序と言葉で説明する。専門用語は使う前に定義する。
3. **まとめ（次に何をするか）**: この教材を終えた後、学習者が次に何をすればよいかを明示する。「初心者が次に何をすればいいか分からない」状態はVeto対象になる。

## やってよいこと

- factsに書かれた事実を、比喩や具体例を使って初心者向けに噛み砕く。
- 専門用語に平易な言い換えを添える。
- 学習目標（`metadata.yaml`の`learning_objective`）に沿って説明の重点を調整する。

## やってはいけないこと

- factsにない情報を追加する（推測・一般論・「よくある話」を根拠なく書く）。
- 実機確認のないUI操作手順を書く（それはlessonではなくworkshopの領域であり、かつ実機根拠が必須）。
- 存在しないボタン名・メニュー名を使う。
- Windows前提と矛盾する説明をする（例: Mac特有のショートカットや手順）。
- 複数の解釈が可能な曖昧な説明をする。

## 用語導入の自然さ

初出の専門用語（例: リポジトリ、コミット）は、日常語で言い換えるか、身近な比喩で説明してから使う。以降は同じ用語を一貫して使い、別の言い方に揺らさない。

## レビュー観点との対応

`eval/criteria/lesson.criteria.md` の評価観点（初心者理解度・Facts忠実性・説明順序・過不足のなさ・用語導入の自然さ）を執筆時から意識すること。書いてからチェックするのではなく、書きながらこの観点を満たすようにする。
